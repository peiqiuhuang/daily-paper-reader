## 摘要
生成式检索已成为工业搜索、推荐和广告系统中日益广泛采用的范式，带来了显著的线上收益。现有的大多数工作将用户行为序列与大语言模型（LLMs）相结合来建模用户偏好。在实践中，特征工程对于模型有效性仍然至关重要，但其复杂性减慢了离线迭代速度，并使在线部署变得沉重且难以复用，且所有这些都处于紧张的在线延迟预算之下。根本原因在于特征处理逻辑与模型架构之间的紧密耦合，每次特征变更都会触及训练和推理代码，且难以在不同场景间复用。为了打破这种耦合，我们提出了 Prompt Generation (PG)，这是一个高级分词器和配置驱动的框架，通过两个声明式 JSON 文件将特征处理逻辑与模型架构解耦。这两个文件作为离线训练和在线推理的唯一事实来源，确保了两个阶段之间的特征一致性。PG 将特征分为四种类型，并使用三个可组合的处理组件来组装和压缩异构特征，在三个层面实现了加速：(1) 快速训练迭代：特征实验仅需修改配置，并内置了针对超长序列的 token 压缩功能；(2) 快速部署：新场景只需符合 PG 模式并接入通用流水线，无需针对特定场景进行工程开发；(3) 快速在线推理：引擎对标准化配置应用统一优化，将 PG 的开销降低到可以忽略不计的水平。PG 已在淘宝搜索部署，在线 A/B 测试显示交易量增长 +0.47%，GMV 增长 +0.51%，具有统计学显著性，并已作为生成式检索的迭代框架应用于多个淘宝搜索和推荐团队。

## Abstract
Generative retrieval has become an increasingly adopted paradigm for industrial search, recommendation, and advertising systems, delivering significant online gains. Most existing work combines user behavior sequences with large language models (LLMs) to model user preferences. In practice, feature engineering remains critical to model effectiveness, yet its complexity slows offline iteration and makes online deployment heavy and hard to reuse, all under tight online latency budgets. The root cause is a tight coupling between feature-processing logic and model architecture, where every feature change touches the training and serving code and resists reuse across scenarios. To break this coupling, we present Prompt Generation (PG), a high-level tokenizer and configuration-driven framework that decouples feature-processing logic from model architecture through two declarative JSON files, which serve as the single source of truth for both offline training and online serving, ensuring feature consistency across the two stages. Organizing features under four types with three composable processing components to assemble and compress heterogeneous features, PG delivers acceleration at three levels: (1)fast training iteration: feature experiments require only configuration changes, with built-in token compression for ultra-long sequences; (2)fast deployment: a new scenario only needs to conform to the PG schema and plug into a universal pipeline, with no scenario-specific engineering; (3)fast online inference: engine applies unified optimizations over the standardized configuration, reducing PG's overhead to a negligible level. PG has been deployed on Taobao Search with statistically significant online A/B uplifts of +0.47% in transaction count and +0.51% in GMV, and has been applied across multiple Taobao search and recommendation teams as the iteration framework for generative retrieval.

*   **核心问题**：在工业级搜索和推荐系统中，生成式检索（基于LLM直接生成目标ID）已成为趋势，但其**特征处理逻辑与模型架构之间存在严重的紧耦合**。
*   **研究背景**：传统的特征工程需要频繁修改训练和推理代码，导致离线迭代慢、在线部署重、且难以在不同场景间复用。此外，离线训练与在线推理之间的“特征不一致（Training-Serving Skew）”也是影响模型效果的关键痛点。
*   **整体含义**：PG框架通过声明式配置（JSON）将特征逻辑从模型中剥离，实现了一套配置驱动的通用流水线，旨在提升迭代效率、确保离在线一致性并优化推理延迟。

### 2. 论文提出的方法论
*   **核心思想**：采用“配置驱动”的哲学，通过两个JSON文件（`prompt_feature.json` 和 `prompt_template.json`）定义特征处理全流程，使其成为离在线唯一的“事实来源”。
*   **关键技术细节**：
    *   **四种特征类型**：文本（Text）、嵌入（Embedding）、组合（Combo）、序列（Sequence）。
    *   **三个可组合组件**：
        1.  **预处理器（Preprocessors）**：对文本进行映射（Mapping）或分桶（Bucketize）。
        2.  **投影器（Projectors）**：通过MLP将外部Embedding对齐到LLM的隐藏层维度。
        3.  **合并器（Mergers）**：提供Mean、Sum、Concat-MLP等策略，将多个特征或Token压缩为固定长度，解决长序列导致的计算开销。
    *   **算法流程**：配置解析 -> 特征提取 -> 模板集成（占位符机制） -> 分词 -> 类型特定处理（如MLP投影） -> 嵌入组装（将处理后的特征插入LLM输入序列）。
    *   **事件追踪（Event Tracking）**：在线记录原始特征并回流至离线，确保训练数据与推理现场完全一致。

### 6. 论文的主要结论与发现
*   **无通用特征配方**：不同场景（搜索 vs. 推荐）的最优特征组合不同，证明了灵活配置框架的必要性。
*   **异构处理至关重要**：文本特征对压缩（Mean-pool）容忍度高，但语义ID（SID）若过度压缩会严重损害效果。
*   **Mean-pool是强力默认项**：在固定Token预算下，简单的均值合并往往优于复杂的MLP或Attention合并。
*   **业务价值**：在淘宝搜索上线后，**交易笔数提升0.47%，GMV提升0.51%**；在店铺搜索中交易笔数提升达4.01%。

### 7. 优点
*   **解耦彻底**：真正实现了“特征工程不改代码”，极大降低了算法工程师与工程团队的协作成本。
*   **极致优化**：针对LLM推理中的Token碎片化问题，通过算子融合、C++索引构建、内存池复用等手段，将PG框架的额外开销降至极低。
*   **支持超长序列**：通过Token压缩技术，使模型能够处理原始长度达45K的超长行为序列。
*   **自动化潜力**：框架与AutoResearch（自动实验代理）结合，实现了特征工程的自动迭代。

### 8. 不足与局限
*   **领域局限**：实验主要集中在电商领域，在内容推荐（如短视频）或社交媒体广告等领域的泛化性有待进一步验证。
*   **特征类型受限**：目前主要支持文本和向量，尚未原生支持图像/视频原始像素输入（虽计划作为未来工作）。
*   **合并器表达力**：现有的合并器（Mean/Sum/MLP）较为简单，对于需要复杂跨特征交互（Cross-feature interaction）的场景可能建模不足。
