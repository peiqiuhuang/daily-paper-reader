## 摘要
生成优化利用大语言模型（LLMs）根据执行反馈迭代改进工件（如代码、工作流或提示词）。这是构建自我改进智能体的一种极具前景的方法，但在实践中仍然非常脆弱：尽管研究活跃，但在被调查的智能体中，只有 9% 使用了任何形式的自动优化。我们认为，这种脆弱性源于为了建立学习循环，工程师必须做出“隐藏”的设计选择：优化器可以编辑什么，以及在每次更新时提供什么样的“正确”学习证据？我们研究了影响大多数应用的三个因素：初始工件、执行轨迹的信用时界（credit horizon），以及将试错过程批处理为学习证据。通过在 MLAgentBench、Atari 和 BigBench Extra Hard 中的案例研究，我们发现这些设计决策可以决定生成优化是否成功，但在之前的研究中很少被明确说明。不同的初始工件决定了 MLAgentBench 中哪些解决方案是可达的，截断的轨迹仍能改进 Atari 智能体，而更大的小批量（minibatches）并不能单调地提高 BBEH 上的泛化能力。我们的结论是，缺乏一种简单、通用的跨领域建立学习循环的方法是其产品化和采用的主要障碍。我们为做出这些选择提供了实践指导。

## Abstract
Generative optimization uses large language models (LLMs) to iteratively improve artifacts (such as code, workflows or prompts) using execution feedback. It is a promising approach to building self-improving agents, yet in practice remains brittle: despite active research, only 9% of surveyed agents used any automated optimization. We argue that this brittleness arises because, to set up a learning loop, an engineer must make ``hidden'' design choices: What can the optimizer edit and what is the "right" learning evidence to provide at each update? We investigate three factors that affect most applications: the starting artifact, the credit horizon for execution traces, and batching trials and errors into learning evidence. Through case studies in MLAgentBench, Atari, and BigBench Extra Hard, we find that these design decisions can determine whether generative optimization succeeds, yet they are rarely made explicit in prior work. Different starting artifacts determine which solutions are reachable in MLAgentBench, truncated traces can still improve Atari agents, and larger minibatches do not monotonically improve generalization on BBEH. We conclude that the lack of a simple, universal way to set up learning loops across domains is a major hurdle for productionization and adoption. We provide practical guidance for making these choices.

#### 1. 核心问题与整体含义
论文探讨了**生成优化（Generative Optimization）**在实际应用中面临的瓶颈。生成优化是指利用大语言模型（LLM）作为优化器，通过“执行-反馈-修改”的迭代循环来改进代码、工作流或提示词。尽管该技术在研究中表现出自我改进的潜力，但在生产环境中的采用率极低（仅 9%）。作者认为，这种脆弱性源于建立“学习循环”（Learning Loop）时必须面对的三个“隐藏”设计决策：**初始工件的选择、信用时界的定义以及经验批处理的规模**。

#### 2. 论文提出的方法论
论文将生成优化的学习循环形式化，并重点分析了三个关键技术维度：
*   **初始工件（Starting Artifact）**：定义优化器的起点。包括代码结构（单函数 vs. 模块化多函数）、文档说明和初始实现。这决定了优化搜索空间的边界。
*   **信用时界（Credit Horizon）**：在多步任务中，决定向 LLM 提供多长的执行轨迹。是基于每一步的即时反馈（短时界）还是完整任务结束后的累积反馈（长时界）进行更新。
*   **经验批处理（Experience Batching）**：类似于深度学习中的 Batch Size。决定在一次优化更新中，将多少个独立试验的（输入、输出、反馈）三元组放入 LLM 的上下文。
*   **技术框架**：实验主要基于 **Trace** 框架实现，该框架将工作流表示为计算图，允许 LLM 针对特定节点进行基于反馈的更新。

#### 3. 实验设计
论文通过三个跨领域的案例研究来验证上述决策的影响：

*   **Big-Bench Extra Hard (BBEH)（提示词优化）**：任务是优化 LLM 的提示词和后处理代码。对比了 Batch Size 为 1、3、5 时的泛化性能。
*   **Benchmark 与对比方法**：使用了 Kaggle 排行榜百分位、人类玩家得分、以及传统的深度强化学习方法（DQN, PPO）作为基准。

#### 6. 论文的主要结论与发现
*   **初始工件决定可达性**：在 ML 任务中，模块化设计在某些任务上优于单函数，但在另一些任务上则相反。初始化的微小差异会显著改变最终性能。
